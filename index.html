账号准备

先把账号和验证方式准备好

优先使用公司指定账号。如果公司尚未统一开通，可先了解个人账号准备流程。

A Gmail/邮箱

  • 用于接收 ChatGPT 注册验证码。
  • 可使用 Gmail，也可使用其他常用邮箱。
  • 建议绑定手机号和备用邮箱，方便找回。

B Apple Account

  • 仅在需要通过 App Store 下载移动端时使用。
  • Apple 官方称 Apple ID 现为 Apple Account。
  • 地区、付款信息按 Apple 官方规则填写，不做规避指导。

C 手机号

  • 用于账号安全验证或找回。
  • 不要使用他人手机号长期绑定。
  • 验证码不要转发给任何人。

D 公司申请

  • 部门确有使用需求时，由主管统一申请。
  • 说明使用场景、人数、周期和数据类型。
  • 涉及敏感数据先咨询 IT/法务。

Codex工作空间

先选好项目文件夹，再开始交给 Codex 做事

工作空间可以理解为 Codex 的“办公桌”。不同项目放在不同工作空间里，文件边界、任务记录和审批会更清楚。

设置沙盒

沙盒是 Codex 被允许读写的项目文件夹范围。它能降低误操作风险，详细权限说明请看后面的“功能模块详解 - 权限管理”。

创建工作空间

可以创建多个项目文件夹作为工作空间，让不同任务分开处理。多个任务可并行执行，常见状态包括：进行中、等待批准、完成。

项目归属对话

新建对话时可以选择它属于哪个项目。选择 不使用项目 时，对话会归属到普通对话区域，适合通用 AI 问答、文案草稿、资料解释等任务。

功能模块详解

这一部分先帮同事认识 Codex 桌面应用里最常用的按钮。先会找对话、会管权限、会看额度，再开始做任务，会稳很多。

搜索历史对话

  • 点击左侧菜单栏的 搜索，或使用快捷键 Ctrl+G 打开对话搜索。
  • 可以搜索所有历史对话，但目前主要搜索 对话标题，不是逐字搜索对话内容。

对话重命名与归档

  • 双击对话 可以重命名，建议用“部门 + 任务 + 日期”命名。
  • 对话太多时，点击对话后面的归档按钮。归档后会从列表移除，可在设置里的已归档对话中取消归档恢复。

权限管理

  • 对话框下方通常有权限下拉选项：请求批准、替我审批、完全访问权限。
  • 请求批准：Codex 对沙箱，也就是当前项目文件夹，有读写权限；不能修改沙箱外文件，且默认禁止联网。需要访问外部文件或联网时，会发起提权并等待人工审批。
  • 替我审批：Codex 会调用模型对提权操作做安全审查。低风险可自动放行，高风险，例如删除文件，仍需要人工审核。
  • 完全访问权限：会无视沙箱限制，可以在电脑上进行更广泛操作。只建议 IT 或明确受控场景使用。

模型选择

  • 推理强度：按任务复杂度选择低、中、高、超高；强度越高，Token 消耗越多。
  • 模型切换：可按界面可用项选择 GPT-5.5、GPT-5.4、GPT-5.4-Mini 等模型。
  • 速度：标准/快速。快速通常响应更快，但可能增加约 1.5 倍 Token 消耗。

剩余额度

  • 在左下角 设置 查看剩余额度。
  • 通常会有每 5 小时小额限额和周限额。任意一个达到上限，都可能暂时无法继续使用。
  • 额度紧张时，可考虑切换更轻量模型，或把任务拆小。

增加参考资料

  • 需要增加图片或文件作为参考时，点击对话框左下角的加号 +。
  • 上传前先检查是否包含客户信息、合同金额、员工隐私等敏感内容。

Codex控制与引导

Codex 有个非常强大的功能，就是 Steer 控制、引导。
  • 引导：当我们发现在 AI 执行过程中，理解错了我们的意思，就不应该让它继续执行了，此时可以人工干预。
  • 跟进行为：我们可以修改跟进行为，在 Codex 运行/思考时将后续操作加入队列，或引导当前运行，按下 Ctrl+Enter 可来回切换跟进和引导。

Codex计划模式

  • 计划模式：普通模式下，你描述需求，AI 直接开始写代码；而计划模式如下：
  • 你的需求 -> AI 提出澄清问题 -> 生成详细计划 -> 你确认 -> AI 执行
  • 该模式适用于：1、首次搭建项目；2、一些大规模的文件/代码改动和重构；3、复杂 Bug 修复；4、多步骤任务。
  • 批注模式：右侧浏览器查看模式下，点击注释，可以分区域标注。回车即可执行注释内容。

Codex记忆系统

开启一个新的对话，就进入一个全新的上下文环境，AI 完全不记得之前发生了什么，甚至对整个项目的记忆都是空白的。当项目变得复杂之后，每次对话都要重新给 AI 提示一遍项目背景，或者让 AI 自己再读一遍代码，就是非常低效的工作方式，还巨消耗 Token。

Codex增加记忆系统

  • 通用方式：在项目根目录创建 AGENTS.md 文件（注意文件名别拼写错误，区分大小写）。
  • 全局方式：Codex 的设置 => 个性化 => 自定义指令。

通用方式

在当前项目的根目录下创建 AGENTS.md 文件，并增加基本信息，注意这个 AGENTS.md 文件只对当前项目生效。例如：

  • Codex 简介的内容已固定，后续维护页面时不再更改该简介文案。
  • 后续入门使用模块新增的图片、卡片样式和尺寸，应与现有页面保持一致。

保存之后 Codex 会自动读取 AGENTS.md 文件的规则，然后把里面的内容自动作为上下文带入对话。

也可以在对话中让 Codex 自动归纳整理 AGENTS.md 文件的内容，然后再人工微调。

通过当前项目，把你学到的关于项目的信息保存到 AGENTS.md 文件

Q：AGENTS.md 文件是什么？

A：写给 Codex 的项目规则文件。

Q：AGENTS.md 应该写什么？

A：写工作约定、验证命令、风险边界，不要塞长篇文档。

AGENTS.md 不是越长越好。它应该写 Codex 做事前必须知道的规则，而不是把所有项目文档复制进去。

适合写进去的内容
  • 项目使用的包管理器、测试命令、构建命令。
  • 修改代码后必须执行的验证步骤。
  • 不能触碰的目录、密钥、生成文件或外部系统。
  • 团队约定的代码风格和提交前检查。
  • 特定子目录的例外规则，例如后端、前端、脚本目录各自的验证命令。
不适合写进去的内容
  • 大段产品背景、历史会议纪要、完整 API 文档。
  • 会频繁变化的临时任务列表。
  • 密钥、账号、Token 或只能给人看的内部凭证。
  • 可以通过命令自动发现的信息，例如完整文件树。

进阶使用

把 Codex 从“会用”变成“可复用的工作流”

进阶阶段重点是标准化：同类任务用同一套提示词、同一套检查清单、同一套安全边界。

模板化

把常用任务写成固定模板：背景、输入、输出格式、风险提示、验收标准。

合规化

建立脱敏规则和人工复核机制，敏感数据默认不上传、不外发、不自动提交。

自动化

将稳定任务做成提醒、检查或批处理流程，先小范围试点，再扩大使用。

团队化

由 IT 维护统一入口、插件、Skills 和说明文档，各部门沉淀自己的业务模板。

进阶原则：越自动化，越要明确权限、日志、复核人和回滚方式。不要让 Codex 直接处理你自己都没有确认过的业务数据。
